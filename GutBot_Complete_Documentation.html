1. Project Overview & Purpose

GutBot is a production-grade AI chatbot built for IOM BioWorks. It allows patients and clinicians to upload medical PDF documents — gut-health reports, microbiome analyses, blood panels, discharge summaries — and ask natural-language questions about them. The system retrieves the most relevant passages from the document and passes them to a large-language model (AWS Bedrock) to generate a grounded, cited answer.

What makes GutBot different from a generic chatbot

  • Document-grounded answers: Every response is rooted in passages from the user's own uploaded PDF, not generic web knowledge.
  • Per-user session isolation: Each browser tab has a unique conversationId. Documents uploaded in one session are completely invisible to every other session.
  • Medical-only gate: A multi-signal heuristic validator rejects non-medical PDFs before they enter the processing pipeline — no invoices, legal contracts, or code files can be indexed.
  • Smart query routing: An inference engine classifies every question before touching FAISS. Greetings, out-of-domain questions, and general medical queries are answered directly by the LLM without a vector search, saving latency and cost.

Primary use cases

  • Patient wants to understand their IOM SEnS / IBS microbiome report
  • Clinician wants to quickly query a discharge summary or blood panel
  • Health professional comparing two lab reports from the same patient

Session isolation design

conversation_indices = { "uuid-alice-session": { "documents": { "doc-uuid-1": { chunks: [...], filename: "report.pdf", selected: True }, }, "combined_index": <faiss.IndexFlatL2>, "doc_to_chunk_mapping": { 0: ("doc-uuid-1", 0), 1: ("doc-uuid-1", 1), ... } }, "uuid-bob-session": { ... }, ← completely separate }
Critical Architecture Constraint
Gunicorn must run with exactly 1 worker process. conversation_indices is a plain Python dictionary in process memory. Multiple Gunicorn workers would be separate OS processes with separate memory — User A's FAISS index would silently disappear when User B's request lands on a different worker. Concurrency is handled by gevent coroutines (non-blocking I/O) within the single worker.

4. PDF Upload & Processing Pipeline

When a user uploads a PDF, the backend executes a strict 7-step pipeline before the document becomes searchable.

Processing steps

  1. File validation — must be .pdf extension, ≤ 50 MB, non-empty filename
  2. Filename sanitisation — werkzeug.secure_filename() strips path traversal characters (../, /etc/passwd, etc.)
  3. Text extraction — PyPDF2 reads every page; rejects blank / corrupted PDFs
  4. Medical document validation — heuristic scorer (see Section 5) rejects non-medical content; file is deleted from disk if rejected
  5. Chunking — text is split into 500-character chunks with no overlap
  6. Embedding generation — SentenceTransformers encodes all chunks into 384-dimensional vectors
  7. FAISS indexing — vectors are added to the conversation's IndexFlatL2; a global chunk-to-document mapping is updated

Upload endpoint — request & response

POST /upload
Content-Type: multipart/form-data

Fields:
  file          (File)    — the PDF

5. Medical Document Validation Layer

NEW FEATURE

backend/medical_doc_validator.py — Implemented as a standalone class with zero dependencies beyond the Python standard library. It runs synchronously in the upload pipeline with no LLM calls, no network requests, and no disk I/O.

Design philosophy

A multi-signal heuristic scoring approach is used. Each signal fires independently and contributes a fixed weight to the total score. A document must score ≥ 4 to be accepted. This prevents gaming (a single medical word is insufficient) while still accepting diverse real-world document formats.

Scoring signals

#SignalTrigger conditionWeight
1 Lab values with units Regex matches numbers followed by medical units (mg/dL, mmol/L, IU/L, etc.) +3 (≥2 hits) / +1 (1 hit)
2 Reference / normal ranges Patterns like normal: 4.0–10.0, ref range 70–99 +3
3 Patient / sample identifier Patient ID, MRN, Date of Birth, Accession No, Collection Date, etc. +3
4 Report header keyword Lab Report, Discharge Summary, Microbiome Report, IOM Report, etc. +4
5 Microbiome / gut-health terms Firmicutes, Lactobacillus, dysbiosis, SCFA, IBS score, GutHeal, etc. +4 (≥2 hits) / +2 (1 hit)
6 Medical test abbreviations CBC, HbA1c, TSH, WBC, HDL, LDL, eGFR, CRP, INR, etc. +2 (≥2 hits) / +1 (1 hit)
7 Medical keyword density 160+ keywords: diagnosis, blood, glucose, cholesterol, hospital, etc. +1 each, capped at +6
8 Non-medical domain patterns Invoice numbers, GST codes, legal clauses (whereas, the parties), source code (def func()) −3 each

Documents accepted (examples)

  • IOM SEnS / IBS microbiome kit reports
  • Blood panel reports (CBC, CMP, HbA1c, lipid profile)
  • Lab test results with reference ranges
  • Discharge summaries and clinical notes
  • Prescriptions with medication dosages
  • Radiology reports
  • Health certificates and medical assessments
  • Medical research papers

Documents rejected (examples)

  • Invoices and purchase orders
  • Legal contracts and NDAs
  • Source code files exported to PDF
  • Business presentations (low medical density)
  • CVs and resumes

Analysis window

Only the first 8,000 characters are analysed — enough to cover the cover page and first section of any real-world document. This keeps validation under 1 ms regardless of PDF length.

No LLM Cost
The validator runs entirely with Python regex. Zero API calls are made for rejected documents. Rejected PDFs are also deleted from disk immediately to avoid orphaned files.

6. RAG Chat Engine

What is RAG?

Retrieval-Augmented Generation (RAG) is a technique where, before calling the LLM, the system first retrieves relevant passages from a knowledge base (here, the user's uploaded PDFs) and injects them as context into the LLM prompt. This grounds the answer in real document content and prevents hallucination.

FAISS vector search

  • Index type: faiss.IndexFlatL2 — exact L2 (Euclidean) distance search, no approximation
  • Embedding model: all-MiniLM-L6-v2 — 384 dimensions, excellent semantic understanding
  • Top-k: k = min(5, total_chunks) — retrieves the 5 most relevant chunks
  • Each chunk is 500 characters of raw extracted text
  • Source file is tracked for each chunk, enabling citations in the LLM prompt

Context assembly

Retrieved chunks are assembled into a structured context block:

[From: blood_report.pdf]
Patient: John Doe  DOB: 12/03/1980
Haemoglobin: 11.2 g/dL  (Reference: 13.0–17.0)
...

---

[From: gut_report.pdf]
IBS Score: 72/100  Firmicutes:Bacteroidetes ratio: 1.8
...

LLM prompt structure

The context + question are combined into a structured prompt sent to Bedrock. The prompt instructs the model to:

  1. Answer only from the provided context
  2. Cite which document a finding comes from
  3. Say "I don't have that information in your documents" when the answer isn't in the context
  4. Maintain conversational continuity using the last 10 messages of chatHistory

7. Inference Engine & Query Routing

Before any vector search, every chat message is classified by InferenceEngine.classify() (in llm_config.py). This routing step prevents wasteful FAISS searches for questions that have nothing to do with the uploaded documents.

Query strategies

StrategyTriggers onAction
greeting "Hi", "Hello", "How are you?" LLM only — no FAISS search
out_of_domain Non-health topics (politics, sports, coding, etc.) LLM-only polite redirect — no FAISS search
general_medical (no doc) General health question, no document uploaded LLM answers from training data — no FAISS search
document_query Question clearly about the uploaded report FAISS search → context → LLM
hybrid Combines personal report data + general medical knowledge FAISS search → context → LLM
follow_up Short pronoun-based reply to prior message ("What does that mean?") FAISS search with prior context → LLM

The engine also uses chat_history to disambiguate follow-up questions — if the previous exchange was about a specific test result, a follow-up like "Is that normal?" is correctly treated as a document query.

8. Security Guard

All user messages pass through SecurityGuard.check() before any processing. It detects prompt injection, jailbreak attempts, and harmful content.

  • Runs before FAISS search and before any LLM call
  • Returns { "safe": false } for flagged input — request is answered with a soft refusal
  • Returns a sanitised version of the message when input is clean
  • Flagged inputs are logged server-side with the conversation ID for audit
  • The soft refusal response is indistinguishable from a normal response to the user (no error code), preventing information leakage about the filter

10. Conversation Isolation

Every browser tab generates a unique conversationId (UUID v4) on page load. All backend operations are keyed by this ID.

  • User A's FAISS index, document chunks, and chat history are in a separate dict key from User B's
  • Conversations cannot read, write, or interfere with each other
  • The isolation can be verified at runtime via POST /debug/isolation-check
  • Conversations persist until the Gunicorn worker is restarted (e.g., after 500 requests via max_requests) — there is intentionally no cross-session persistence
Memory Growth Note
conversation_indices has no TTL eviction policy. Each conversation occupies RAM proportional to the total size of its document chunks. On a t3.large (8 GB) with the MemoryMax=4G systemd ceiling, ~200–300 simultaneous live conversations is a safe upper bound before the worker is automatically killed and restarted by systemd. The max_requests=500 Gunicorn setting provides a periodic memory-flush.

  7. Set .env file permissions to 600
  8. Start Gunicorn service and reload Nginx

SSH safety under load

The systemd service is configured to protect SSH access even under full load:

  • CPUQuota=75% — GutBot cannot consume more than 75% CPU; SSH always has headroom
  • MemoryMax=4G (adjust per instance) — hard RAM ceiling
  • MemorySwapMax=0 — disables swap; process is killed rather than degrading indefinitely
  • OOMScoreAdjust=300 — GutBot is killed by the OOM killer before sshd
  • Restart=on-failure — auto-restarts within 5 seconds after a crash
  • WantedBy=multi-user.target — auto-starts after every reboot

    "Resource": "arn:aws:bedrock:ap-south-1::foundation-model/*"
  }]

17. Bug Fixes Implemented

The following bugs were identified and fixed during the production audit:

Bug 1 — Chat history never sent to backend CRITICAL

File: frontend/src/App.jsx
Problem: The /chat payload used the key files instead of chatHistory. The backend reads data.get("chatHistory", []). This meant every message was treated as a standalone question — follow-up questions and conversation context never worked at all.

// BEFORE (broken):
await apiClient.post("/chat", {
  message: message.trim(),
  conversationId: currentConversationId,
  files: currentConversation?.files || [],   // ← wrong key
});

// AFTER (fixed):
const chatHistory = (currentConversation?.messages || [])
  .filter((m) => m.role === "user" || m.role === "assistant")
  .slice(-10)
  .map((m) => ({ role: m.role, content: m.content }));

await apiClient.post("/chat", {
  message: message.trim(),
  conversationId: currentConversationId,
  chatHistory,    // ← correct key, correct structure
});

Bug 2 — Wrong port throughout frontend HIGH

Files: .env, .env.example, vite.config.js, services/apiClient.js, utils/api.js
Problem: All fallback URLs used port 5000. The Flask backend runs on port 8000. Any time VITE_API_URL was unset, all API calls silently went to the wrong port.

// BEFORE (broken in 5 files):
"http://localhost:5000"

// AFTER (fixed):
"http://localhost:8000"

Bug 3 — Production API 404 for all requests HIGH

File: ec2-files/nginx-gutbot
Problem: The production frontend sets VITE_API_URL=/api, so all calls become /api/chat, /api/upload, etc. The original Nginx config only matched /chat and /upload — every production request returned 404 or fell through to the React SPA.

# BEFORE (broken):
location /upload { proxy_pass http://gutbot_backend; }
location /chat   { proxy_pass http://gutbot_backend; }

# AFTER (handles both /upload and /api/upload):
location ~ ^(/api)?/upload$ {
    rewrite ^/api(/.*) $1 break;
    proxy_pass http://gutbot_backend;
}
location ~ ^(/api)?/(chat|documents|manage-documents|remove-document|debug) {
    rewrite ^/api(/.*) $1 break;
    proxy_pass http://gutbot_backend;
}

Bug 4 — DocumentManager error retry wiped all app state MEDIUM

File: frontend/src/components/DocumentManager.jsx
Problem: When fetching the document list failed, the retry button called window.location.reload() — a full page reload that wiped all conversations, messages, and files from React state and localStorage was never committed back.

// BEFORE (broken):
const handleRetryFetch = () => {
  setError(null);
  window.location.reload();   // ← nukes everything
};

// AFTER (just refetches documents):
const fetchDocumentsRef = useRef(null);
fetchDocumentsRef.current = fetchDocuments;

const handleRetryFetch = useCallback(() => {
  setError(null);
  fetchDocumentsRef.current?.();   // ← only refetches document list
}, []);

Bug 5 — start.sh pointed to deleted file and wrong port MEDIUM

File: backend/start.sh
Problem: The launch script called gunicorn app_enhanced:app --port 5000. app_enhanced.py had been deleted; port 5000 was wrong. The backend could not be started via this script.

# BEFORE (broken):
gunicorn app_enhanced:app --bind 0.0.0.0:5000

# AFTER (fixed):
gunicorn app:app --bind 127.0.0.1:8000 \
  --worker-class gevent --workers 1 \
  -c gunicorn.conf.py

Bug 6 — Duplicate null check in DocumentManager LOW

File: frontend/src/components/DocumentManager.jsx
Problem: The early-return if (documents.length === 0 && !isLoading) return null; appeared twice — once at line 143 and again at line 160. The second check was dead code after JSX refactoring.

18. Security Hardening

Fix 1 — Path traversal vulnerability SECURITY

File: backend/app.py — /upload
Problem: file.filename was used directly in os.path.join(UPLOAD_FOLDER, file.filename). An attacker could upload a file named ../../etc/passwd or ../app.py to overwrite arbitrary files on the server.

# BEFORE (vulnerable):
filepath = os.path.join(UPLOAD_FOLDER, file.filename)

# AFTER (safe):
from werkzeug.utils import secure_filename
safe_name = secure_filename(file.filename)
if not safe_name:
    return jsonify({"error": "Invalid filename"}), 400
filepath = os.path.join(UPLOAD_FOLDER, safe_name)

Fix 2 — CORS production whitelist silently overridden SECURITY

File: backend/app.py — after_request hook
Problem: A global @app.after_request hook unconditionally added Access-Control-Allow-Origin: * to every response, overriding the carefully built production whitelist in flask-cors. Any origin could read the API in production.

# BEFORE (overrides CORS in production):
@app.after_request
def after_request(response):
    response.headers["Access-Control-Allow-Origin"] = "*"   # ← always
    ...

# AFTER (only in development):
@app.after_request
def after_request(response):
    if os.getenv("FLASK_ENV") != "production":
        response.headers["Access-Control-Allow-Origin"] = "*"
    # Production: flask-cors already set the correct whitelist header
    ...

Fix 3 — Nginx Security Headers SECURITY

The Nginx site config adds the following headers to all responses:

  • X-Content-Type-Options: nosniff — prevents MIME-type sniffing
  • X-Frame-Options: DENY — prevents clickjacking iframes
  • X-XSS-Protection: 1; mode=block — legacy XSS filter in older browsers
  • Referrer-Policy: no-referrer — prevents leaking URL in Referer header

Fix 4 — File permissions on .env SECURITY

The deploy script sets chmod 600 backend/.env — readable only by the owning user (ubuntu), not by other system users or the web server process.

Fix 5 — Gunicorn not exposed to internet

Gunicorn binds to 127.0.0.1:8000 (loopback only). The EC2 Security Group does not open port 8000 to the public. Only Nginx (on port 80/443) is internet-facing, and it rate-limits all requests before forwarding.

Fix 6 — Frontend input sanitisation

apiClient.js strips < and > from all string fields before transmission to prevent reflected XSS via API responses rendered as HTML. The SecurityGuard in the backend provides a second layer of prompt-injection filtering.

20. Known Limitations & Future Work

Limitations in current version

1. In-memory session store (no TTL)

conversation_indices grows unbounded. Each active conversation holds all chunk embeddings in RAM. On a long-running instance, this will accumulate over time. The max_requests=500 Gunicorn setting provides a periodic flush (worker restart), but idle conversations are not evicted between flushes.

Mitigation already in place: max_requests=500, MemoryMax systemd ceiling

Future fix: Add a TTL using threading.Timer or switch to Redis for session storage

2. PDF text extraction quality

PyPDF2 extracts raw text. Scanned PDFs (image-only), PDFs with complex column layouts, or PDFs with non-standard encoding may produce garbled or empty text.

Future fix: Add OCR fallback using pytesseract or AWS Textract for image PDFs

3. No user authentication

There is no login, no access control, and no persistent user accounts. conversationId is generated client-side — anyone who knows a UUID can query that session's documents.

Future fix: Add JWT authentication or AWS Cognito

4. Chunk overlap

Text is split into 500-character non-overlapping chunks. A sentence that straddles a chunk boundary may be split in half, reducing retrieval accuracy for those passages.

Future fix: Use LangChain's RecursiveCharacterTextSplitter with 50–100 character overlap

5. Debug endpoint is public

POST /debug/isolation-check exposes internal session metadata. Acceptable for internal testing, but should be removed or gated behind auth before a public launch.

6. Single region AWS Bedrock

The app is locked to ap-south-1 by default. If Bedrock has an outage in that region, the entire chatbot goes down.

Future fix: Implement a fallback region in llm_config.py

Possible enhancements

  • Streaming responses — Bedrock supports streaming; implementing it would show tokens as they arrive
  • Multi-modal support — accept DOCX, images of reports via AWS Textract
  • Source highlighting — highlight the exact sentence in the PDF that was retrieved
  • Session persistence — save conversation history and documents to DynamoDB or S3
  • Feedback loop — thumbs up/down on responses to fine-tune the retrieval strategy
  • Admin dashboard — view active sessions, document count, request volume
